# GameAuditAgent: 状态机路径执行与覆盖模型 (FSM-based Audit)

## 目标与背景

当前 GameAuditAgent 采用线性的测试计划生成机制（`GeneratePlanTool`）。这虽然直观，但面对 Web 游戏高度网状的交互状态（例如：`主界面 -> 商店 -> 购买确认 -> 余额不足弹出 -> 充值指引 -> 返回主界面`），一维的测试列表极易引发以下问题：
1. **目标遗漏的黑盒效应**：仅测试“主干道”，难以遍历所有必须覆盖的边缘状态。
2. **意外流转导致的脱轨**：由于突发的运营弹窗或掉线重连框，固定的测试步骤会瞬间失效并中止，导致后面规划好的模块被直接跳过。

为了彻底打消对**“分析和目标可能遗漏”**的担忧，我们提议将测试模型从“线性列表用例（Linear Cases）”进化为**基于有限状态机（Finite State Machine, FSM）的图探索模型**。以图（Graph）作为目标规划和执行指引，确保核心状态节点能够被 100% 度量与覆盖。

## 技术方案

### 1. 从 PDF 到状态图的提取 (FSM Graph Generation)
在阅读完 PDF 玩法指南后（Phase 1），大语言模型不再仅输出一条条用例记录，而是输出一个有向图（Directed Graph）的数据结构。
*   **状态节点 (Nodes)**: 游戏中独立的页面、层级视图或固定弹窗（例如：`HomeContext`, `StorePage`, `PaymentModal`, `ErrorAlert`）。
*   **转换边 (Edges)**: 连接节点的操作行为（例如：`click(Buy_Button)`, `wait(10s)`）。
*   **预期断言 (Assertions on Nodes/Edges)**: 当到达特定节点或途经某条边时，必须连带执行的审计规则（如到达 `StorePage` 后必须完成价格列表敏感词检测）。

### 2. 算法驱动的探索与路径生成 (FSM Path Execution)
在（Phase 2）执行期间，GameAuditAgent 使用图层面的遍历策略，取代按顺序往下走：
*   **深度优先搜索 (DFS)**：倾向于一路点击到底（例如体验一套完整的新手流程任务），用于测试纵深核心玩法的连贯生命周期。
*   **广度优先搜索 (BFS)**：快速把当前页面可见的所有功能入口（边）都进退尝试一次，用于挖掘和排查错误链接和无效入口，保证无死角功能覆盖。
*   **反向/负面测试 (Negative Edge Targeting)**：针对不能导致状态变更的恶意操作（比如试图反复快速点击正在转圈 Loading 的按钮）进行强制校验，看是否引起前端崩溃。

### 3. FSM 探察调度器与运行时动态修正 (Dynamic Runtime Correction)
游戏的实际页面可能隐藏着文档未描述的暗线卡点：
*   Agent 在内存中将维护一张 **运行时状态图 (Runtime Metagraph)**。
*   预期外节点发现：当预期执行 `操作A` 会去 `节点B`，却弹出了一个未知 `节点X`（例如防沉迷认证弹窗），Agent 会立即中止原有的遍历指针。将 `节点X` 注册入图中，并计算如何关闭 `节点X` 或妥善绕过它，再回到原状态树。这就保证了后续所有测试目标不会因为单一的阻断而被丢弃（Graceful Recovery）。

### 4. 覆盖率度量与可视化防范未然 (Coverage Analytics)
每次审计结束后，生成的 `AuditReportV1` 将拥有更严谨的空间报告能力：
*   比对 **模型图节点数** 与 **实际运行时触达的节点数**，得出 FSM 节点覆盖率和转换覆率（Coverage %）。
*   利用 Mermaid 在最终报告中画出彩色的状态轨迹图。被点亮的节点为已验证路线；呈**灰色**或**暗色**的节点会在图下醒目标注：“这是 PDF 规定的核心目标，但由于 XXX 阻断在此次测试中遭到遗漏”。
*   人类审查者将对“遗漏比例”一目了然，绝不含糊。

## 影响范围与风险评估

*   **影响模块**: `kubemin_agent/agents/game_audit/models.py` (新增 FSM 结构定义替换单纯的列表用例)、`game_audit_agent.py` 的主循环架构、CLI 展板展现。
*   **风险挑战**: 万一游戏在某个状态陷入了类似 `战斗结束计算->又触发彩蛋战斗` 的死循环，可能会耗尽大量资源与 LLM Token。
*   **缓解措施**: 引入 **Snapshot Tokenization / 历史状态哈希计算**。当 Agent 发现处于同一哈希状态、没有新的图变化，并且已经在这个循环超过 3 次（`Max_Visits_Per_Node`），果断终止当前死路径分支，退出外层，避免 Token 燃烧。

## 验证计划

1.  **单元测试**: 输入一个虚构的“商店买卖 PDF 手册”，验证提示词能否准确抽取 FSM 结构的 JSON（不仅包含 Case，更包含前置上下文与流转箭头）。
2.  **集成实训**: 开启一个有“陷阱页”（点击任意地方都会弹出警告）的 Mock Server 游戏，运行新的 FSM Agent，断言其不仅识别了陷阱页能自动恢复找寻退出按钮，且最终报告精准计算出了页面探索覆盖率。
3.  **人类核对审查**: 通过最终产生的 Markdown 报告，核对 Mermaid 状态连线图与预演时的用户期望是否存在偏离。
